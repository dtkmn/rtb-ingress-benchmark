Methodology

How to read these rankings

This dashboard is intentionally mode-specific. It is designed to clarify tradeoffs, not flatten them into one universal winner board.

Benchmark Modes

Three modes, three different questions

Confirm

Measures combined HTTP handling plus Kafka delivery confirmation. Useful for more conservative ingress behavior where the broker stays in the request path.

HTTP-only

Measures request parsing, validation, filtering, and response handling without Kafka. This is the cleanest read on framework/runtime overhead.

Enqueue

Measures HTTP handling plus client enqueue/fire-and-forget behavior. Faster than confirm, but not equivalent to durable end-to-end delivery.

Fairness Controls

What is held constant

  • Workload shape, service set, CPU budget, memory budget, and Kafka broker budget.
  • Explicit concurrency knobs such as `HTTP_SERVER_WORKERS`, `GOMAXPROCS`, and Quarkus I/O threads.
  • Shared Kafka producer tuning where the client library allows it.
  • Mode-specific rankings instead of one global leaderboard.

Read Carefully

What these rankings do not mean

  • They are not universal backend rankings.
  • They do not replace workload-specific capacity testing on production-like infrastructure.
  • They do not make startup time, memory footprint, and steady-state throughput interchangeable.
  • They do not justify language-war conclusions without looking at bottlenecks and tradeoffs.
